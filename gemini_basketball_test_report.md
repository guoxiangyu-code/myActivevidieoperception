## 三、关键差异分析

### 3.1 Planner 策略差异

| 维度 | Gemini | Qwen |
|------|--------|------|
| **区间选择** | [0, 300]s (从头开始5分钟) | [125, 180]s (基于temporal hint精确定位) |
| **策略理由** | "exact time unknown, use a generous window" | "temporal hints point to 02:05-03:00" |
| **效果** | 覆盖更全面，但发送更多数据 | 更精确，但遗漏了比赛开始到125s的内容 |

Gemini Planner 更保守，选择从视频开头观察完整5分钟，确保不遗漏任何开场得分。Qwen Planner 则直接依赖 temporal hint 定位，虽然精确但假设了 hint 的准确性。

### 3.2 Observer 视频理解能力

| 维度 | Gemini | Qwen |
|------|--------|------|
| **视频输入** | 原生视频字节 (12.41MB clip) | 32 张 JPEG 帧 |
| **信息量** | 完整视频流 + 音频 | 每1.7秒采样一帧，无音频 |
| **球员识别** | ✅ 正确识别杜兰特、罗伯森、威斯布鲁克 | ❌ 未识别球员，仅说"不同球员" |
| **得分类型区分** | ✅ 区分上篮(2分)和三分球 | ❌ 将上篮误判为三分出手 |
| **时间定位精度** | 精确 [157,159]s, [167,169]s (2秒窗口) | 粗略 [125,140]s, [145,160]s, [170,180]s (15秒窗口) |
| **比分追踪** | ✅ 准确 (2-0, 5-0, 8-0) | ❌ 无比分信息 |

### 3.3 核心差距：原生视频 vs 帧采样

```
Gemini (原生视频):
  视频帧 ──────────────────── 连续画面，每个动作都可见
  音频轨 ──────────────────── 解说员实时报分、评论
  ↓
  "威斯布鲁克上篮得2分" → 准确区分投篮类型
  "杜兰特三分命中" → 识别具体球员+投篮方式

Qwen (帧采样):
  帧1 ... 帧2 ... 帧3 ... (每1.7秒一帧)
  ↓
  静态图片无法判断投篮方式 → 看到得分就标记为"三分出手"
  无音频信息 → 无法从解说获取得分类型
```

## 六、结论

### Gemini 2.5 Pro 的优势

1. **原生视频理解**: 直接处理视频流，保留完整时序信息和音频
2. **球员识别**: 准确识别具体球员 (杜兰特、罗伯森、威斯布鲁克)
3. **动作区分**: 区分上篮(2分)和三分球投射
4. **时间定位**: 精确到2秒窗口 vs Qwen的15秒窗口
5. **比分追踪**: 实时追踪比分变化 (2-0, 5-0, 8-0)
6. **速度**: 164s vs 290s (快77%)

### Qwen 在 frames 模式下的局限

1. **帧间信息丢失**: 1.7秒间隔无法捕捉投篮动作细节
2. **无音频**: 缺少解说员的得分类型报告
3. **模糊描述**: 无法识别球员身份，无法区分投篮类型
4. **粗粒度时间戳**: 15秒窗口 vs Gemini的2秒窗口

### 改进方向

- **对 Qwen**: 提高 max_frame 至 128+，或找到支持 video_url 的 API 端点
- **对 Gemini**: 需要解决 yunwu.ai 的 body size 限制，长视频需要压缩版
- **通用**: 在 prompt 中强调区分投篮类型（上篮/中投/三分/罚球）

## 七、技术备注

- Gemini 测试需要先创建压缩版视频（496MB -> 78MB），否则 yunwu.ai 拒绝大请求体
- 压缩参数: `libopenh264 -b:v 300k -vf scale=480:-2 -c:a aac -b:a 48k`
- 压缩版虽然画质降低(480p/300kbps)，但 Gemini 仍能准确识别球员和动作
- 这说明 Gemini 的视频理解能力不完全依赖高分辨率，时序连续性更重要
