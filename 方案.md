## 一、传给 Gemini 2.5 Pro 模型的数据是什么？

在这个 AthenaQA 框架中，Gemini 2.5 Pro 模型在不同角色（Planner / Observer / Reflector-Synthesizer）中被调用，传入的数据各不相同：

### 1. **Planner（规划阶段）** — 纯文本
调用 `plan_replan_model`（即 `gemini-2.5-pro`），传入的是 **纯文本 prompt**：
```python
resp = self.client.models.generate_content(
    model=self.plan_replan_model, 
    contents=prompt  # 纯文本
)
```
Prompt 内容包含：用户 query、视频元数据（时长等）、MCQ 选项（如果有）。**不传视频数据**。

### 2. **Observer（观察阶段）** — 文本 + 视频二进制数据 ⭐ 核心
调用 `execute_model`（即 `gemini-2.5-pro`），传入的是 **文本 prompt + 视频 Part 对象**：
```python
contents = [prompt] + parts  # prompt 是文本，parts 是视频 Part 列表
resp = self.client.models.generate_content(
    model=self.execute_model, 
    contents=contents,
    config=generate_content_config  # 含 media_resolution 设置
)
```

每个视频 Part 的构造方式（`create_video_part` 方法）：
- **视频二进制数据**：直接 `open(video_path, "rb")` 读取整个视频文件的字节，封装为 `Blob(mime_type=mime_type, data=video_data)`
- **视频元数据 `VideoMetadata`**：
  - `fps`：采样帧率（会根据 `max_frame` 限制自动调整）
  - `startOffset` / `endOffset`：时间偏移量（如 `"10.5s"`）
- **`GenerateContentConfig`**：设置 `media_resolution`（`MEDIA_RESOLUTION_LOW` / `MEDIUM` / `HIGH`）

简而言之，**传给 Gemini 的核心数据是：视频文件的原始二进制字节 + 采样/裁剪的元数据参数 + 文本 prompt**。Gemini 原生支持直接理解视频，不需要逐帧抽图。

### 3. **Synthesizer / Reflector（综合/反思阶段）** — 纯文本
传入的是纯文本（evidence 摘要 + query），**不传视频数据**。

## 二、ffmpeg 在这个项目中做什么？

**ffmpeg 用于在 "region" 模式下裁剪视频片段（clip）。** 具体来说：

### 使用场景
当 Planner 决定只观察视频的**特定时间区间**（`load_mode="region"`）时，框架需要：

1. **裁剪视频片段**（`create_video_clip`，在 `avp/video_utils.py` 中）：
   - 使用 ffmpeg 从原始长视频中按时间范围 `[start_time, end_time]` 切出小 clip
   - 命令格式：`ffmpeg -ss HH:MM:SS.mmm -to HH:MM:SS.mmm -i input.mp4 ... output.mp4`
   - 切出的 clip 保存在 `temp_clips/` 目录

2. **重编码全片视频**（`create_reencoded_video_clip`）：
   - 当视频时长 ≥ 900 秒（15分钟）且使用 uniform 模式时
   - 用 ffmpeg 重编码为更小的分辨率/码率，减少传给 Gemini 的数据量
   - 例如 low 模式：缩放到 480p，视频码率 220k，帧率 6fps

3. **Fallback 机制**：
   - 如果 `check_ffmpeg_available()` 返回 False（没装 ffmpeg），框架会回退到使用 Gemini 的 `startOffset`/`endOffset` 元数据方式，把整个视频传过去并指定偏移量
   - 所以 **ffmpeg 不是严格必需的**，但有了它裁剪效率更高、传输数据量更小

### 安装方式
README 中指定的安装方式是通过 conda：
```bash
conda install -c conda-forge ffmpeg
```

## 二、`create_video_part()` — 构造视频 Part 的完整细节

这是最核心的函数（[`main.py` 第 722-833 行](https://github.com/guoxiangyu-code/AthenaQA/blob/c2326a4f2f68d1d9f7e17867378ed30845a909c8/avp/main.py#L722-L833)），逐步拆解：

### 步骤 1：解析视频路径（支持压缩版本优先）

```python
actual_video_path, compressed_path = resolve_video_path(
    video_path,
    prefer_compressed=self.prefer_compressed,  # 默认 True
)
```

如果存在 `compressed_xxx.mp4` 或 `xxx_compressed.mp4`，会自动使用压缩版本以减少上传体积。

### 步骤 2：读取整个视频文件的原始二进制字节

```python
mime_type = get_mime_type(actual_video_path)  # 根据扩展名：.mp4 → "video/mp4"

with open(actual_video_path, "rb") as f:
    video_data = f.read()                     # ← 整个文件一口气读进内存！

blob = Blob(mime_type=mime_type, data=video_data)
```

**关键点**：`video_data` 就是视频文件的**完整二进制字节**（整个 .mp4 文件），不是帧、不是截图、不是 base64——就是原始文件内容。`Blob` 是 Google GenAI SDK 的数据载体类。

### 步骤 3：构建 `VideoMetadata`（告诉 Gemini 如何采样）

```python
video_metadata = None
if fps or start_offset or end_offset:
    kwargs = {}
    
    if fps:
        # 根据 media_resolution 选择最大帧数上限
        if media_resolution == "low":
            max_frame = self.max_frame_low      # 默认 512
        elif media_resolution == "medium":
            max_frame = self.max_frame_medium    # 默认 128
        elif media_resolution == "high":
            max_frame = self.max_frame_high      # 默认 128
        
        # 关键：如果 fps × 时长 > max_frame，自动降低 fps
        if duration and duration > 0:
            expected_frames = fps * duration
            if expected_frames > max_frame:
                adjusted_fps = max_frame / duration
                fps = adjusted_fps    # ← FPS 被钳制！
        
        kwargs["fps"] = fps
    
    if start_offset:
        kwargs["startOffset"] = start_offset    # 如 "10.5s"
    if end_offset:
        kwargs["endOffset"] = end_offset        # 如 "60s"
    
    video_metadata = VideoMetadata(**kwargs)
```

**这段代码的意思是**：你给 Gemini 传了整个视频文件，但通过 `VideoMetadata` 告诉它：
- **`fps`**：请以每秒 X 帧的速率采样（Gemini 服务端按此采样，而非客户端逐帧抽图）
- **`startOffset` / `endOffset`**：只看视频的 [start, end] 时间范围
- **FPS 限制机制**：如果 `fps × 视频时长 > max_frame`，就自动降低 fps。比如一个 1000 秒的视频、fps=1、max_frame=512 → 调整为 fps=0.512

### 步骤 4：组装 `Part` 对象

```python
return Part(inlineData=blob, videoMetadata=video_metadata)
```

最终的 `Part` 对象包含两部分：
| 字段 | 内容 | 作用 |
|------|------|------|
| `inlineData` | `Blob(mime_type="video/mp4", data=<原始字节>)` | 视频的完整二进制文件 |
| `videoMetadata` | `VideoMetadata(fps=X, startOffset="Ys", endOffset="Zs")` | 告诉 Gemini 如何采样/截取 |

> **注意**：`media_resolution`（空间分辨率）**不在 Part 中**，而是放在 `GenerateContentConfig` 中。代码注释明确说明了这点（第 828 行）。

## 三、`infer_on_video()` — 三种视频传入路径

`infer_on_video()` 根据 `watch_cfg.load_mode` 和其他条件，选择不同的视频传入方式。一共有 **三条主路径**：

### 路径 A：Region 模式 + 多区域（ffmpeg 裁剪 clip）

**触发条件**：`load_mode == "region"` 且 `watch_cfg.regions` 有多个区域

```python
for region_idx, (reg_start, reg_end) in enumerate(watch_cfg.regions):
    # 用 ffmpeg 从原视频裁剪出 clip
    clip_path = create_video_clip(video_path, reg_start, reg_end, ...)
    
    if clip_path:
        # clip 裁剪成功 → 把 clip 文件作为 Part（不需要 offset）
        part = self.create_video_part(
            video_path=clip_path,          # ← 传入的是 clip 文件
            fps=fps,
            start_offset=None,             # clip 本身就是那个片段，不需要 offset
            end_offset=None,
            media_resolution=media_res,
            duration_sec=clip_duration,
        )
        parts.append(part)
    else:
        # ffmpeg 不可用/裁剪失败 → 传整个视频 + offset 元数据
        part = self.create_video_part(
            video_path=video_path,         # ← 传入整个原始视频
            fps=fps,
            start_offset=f"{reg_start}s",  # ← 靠 offset 告诉 Gemini 只看这段
            end_offset=f"{reg_end}s",
            media_resolution=media_res,
        )
        parts.append(part)
```

**结果**：多个 Part，每个 Part 对应一个时间区域的视频。

### 路径 B：Region 模式 + 单区域

逻辑类似路径 A，只是只有一个区域。同样先尝试 ffmpeg 裁剪 clip，失败则回退到 offset 方式。

### 路径 C：Uniform 模式（看整个视频）

**触发条件**：`load_mode == "uniform"`

```python
# 特殊优化：如果视频 ≥ 900s（15分钟），尝试重编码压缩
if duration_sec >= 900.0:
    clip_path = create_reencoded_video_clip(
        video_path, start_sec, end_sec,
        scale_width=480,        # 缩到 480p
        video_bitrate="220k",   # 低码率
        frame_rate=6.0,         # 固定 6fps
        crf=32,                 # 高压缩率
    )
    
    if clip_path:
        part = self.create_video_part(clip_path, fps, ...)
    else:
        # 重编码失败 → 传原始视频 + offset
        part = self.create_video_part(video_path, fps, start_offset, end_offset, ...)
else:
    # 视频 < 15 分钟 → 直接传原始视频 + offset
    part = self.create_video_part(video_path, fps, start_offset, end_offset, ...)
```

## 五、三个控制维度的详细解释

| 维度 | 参数 | 在哪里设置 | 作用 | 默认值/范围 |
|------|------|-----------|------|-----------|
| **时间采样率** | `VideoMetadata.fps` | Part 的 videoMetadata | Gemini 服务端每秒取几帧 | Planner 决定，受 max_frame 钳制 |
| **时间范围** | `VideoMetadata.startOffset/endOffset` | Part 的 videoMetadata | 只分析视频的哪个时间段 | 仅在 offset 模式下使用 |
| **空间分辨率** | `GenerateContentConfig.media_resolution` | API 调用的 config | 每帧图像的分辨率/token 数 | LOW/MEDIUM/HIGH |

### FPS 钳制逻辑（很重要）：

```
期望帧数 = fps × 视频时长
如果 期望帧数 > max_frame:
    fps = max_frame / 视频时长
```

例如：
- `fps=1, 时长=1000s, max_frame_low=512` → `1 × 1000 = 1000 > 512` → 调整为 `fps = 512/1000 = 0.512`
- 这意味着 1000 秒视频最多取 512 帧

## 六、为什么叫"原生支持"？

对比传统方案：

| | 传统方案 | AthenaQA / Gemini 原生 |
|--|---------|----------------------|
| 客户端处理 | OpenCV 逐帧抽图 → PIL 编码为 base64 → 每帧作为 image Part | `open(video, "rb").read()` → 整个文件 bytes |
| 传给模型的 | N 张 base64 图片 | 1 个视频 Blob + 采样元数据 |
| 帧采样在哪 | 客户端 (OpenCV/ffmpeg) | Gemini 服务端（根据 VideoMetadata.fps） |
| 时间定位 | 客户端计算帧索引 | Gemini 服务端（根据 startOffset/endOffset） |
| 依赖 | OpenCV, PIL, 视频编解码库 | 只需 `google-genai` SDK |
| 注释原话 | — | `"No OpenCV or video codec dependencies required!"` (第 35 行) |

**本质区别**：框架把采样/解码的责任**全部交给了 Gemini 服务端**。客户端只负责读文件字节 + 告诉服务端"以什么速率采样、看哪段"。

### 关键区别：什么叫"原生"vs"非原生"？

| | 原生视频支持 | 非原生（传统方案） |
|--|------------|----------------|
| **传入格式** | 视频文件二进制字节（mp4/webm等） | 客户端抽帧 → N 张 base64 图片 |
| **帧采样** | 模型服务端自行采样 | 客户端 OpenCV/ffmpeg 抽取 |
| **时序理解** | 模型内部理解帧间关系和运动 | 模型只能看到一系列独立图片 |
| **代表** | Gemini, Qwen3.5 VLM, Emu3.5 | 早期 GPT-4V, LLaVA-Video 等 |

AthenaQA 选择 Gemini 的原因正是因为它是**最早且最成熟**提供原生视频 API（`inlineData` + `VideoMetadata`）的商用模型，客户端零依赖视频编解码库即可完成视频理解。

---

## 一、核心结论：不能直接替换，需要大量改造

AthenaQA 的整个框架**深度绑定了 Gemini 的专属 API 特性**，千问 VL（Qwen-VL）系列和 Gemini 的视频传入方式有**根本性差异**。

---

## 二、具体的绑定点（至少 5 层）

### 绑定点 ①：SDK 层面 — 直接使用 `google-genai` SDK

```python
# main.py 第 62-64 行
from google import genai
from google.genai import types
from google.genai.types import Part, Blob, VideoMetadata
```

AthenaQA 使用的 `Part`、`Blob`、`VideoMetadata`、`GenerateContentConfig` 等类**全部是 Google GenAI SDK 的专有类型**。千问没有这些类。

### 绑定点 ②：视频传入方式 — 原始二进制 Blob vs 逐帧图片

| | Gemini（AthenaQA 现在用的） | 千问 VL 系列 |
|--|--------------------------|-------------|
| **传入格式** | `Blob(mime_type="video/mp4", data=原始字节)` | **必须客户端抽帧** → 传入图片数组 |
| **帧采样** | 服务端根据 `VideoMetadata.fps` 自动采样 | 客户端用 OpenCV/ffmpeg 抽帧 |
| **时间范围** | `VideoMetadata(startOffset="10s", endOffset="60s")` | 客户端自行计算帧索引截取 |

**千问 VL 的典型用法**：
```python
# 千问的方式：必须自己抽帧！
images = [Image.open('frame_1.png'), Image.open('frame_2.png'), ...]
response = model.chat(images=images, query='What is happening?')
```

**Gemini 的方式（AthenaQA 现在用的）**：
```python
# Gemini的方式：直接传视频文件字节
with open("video.mp4", "rb") as f:
    blob = Blob(mime_type="video/mp4", data=f.read())
part = Part(inlineData=blob, videoMetadata=VideoMetadata(fps=1))
```

这是**根本性的架构差异**。AthenaQA 文件头注释明确自豪地写着：
> *"No OpenCV or video codec dependencies required!"*

如果换成千问，就必须引入 OpenCV 等视频解码库，整个视频处理管线要重写。

### 绑定点 ③：`VideoMetadata` 采样控制 — 千问没有等价物

AthenaQA 中 `create_video_part()` 构建的 `VideoMetadata`：

```python
VideoMetadata(
    fps=0.512,           # 告诉 Gemini 服务端每秒采样 0.512 帧
    startOffset="10.5s", # 只看 10.5 秒之后
    endOffset="60s",     # 到 60 秒为止
)
```

这个机制在千问 VL 中**完全不存在**。千问 VL 你传什么帧，它看什么帧——采样控制完全在客户端。

### 绑定点 ④：`GenerateContentConfig.media_resolution` — 千问没有等价物

```python
generate_content_config = types.GenerateContentConfig(
    media_resolution="MEDIA_RESOLUTION_LOW"  # 控制空间分辨率/token 消耗
)
```

这是 Gemini 专有的参数，用来控制每帧图像被编码为多少 token。千问 VL 的空间分辨率由你传入的图片尺寸决定。

### 绑定点 ⑤：Planner 的输出 schema 假设了 Gemini 的参数

Planner 生成的 `WatchConfig` 包含：
```python
@dataclass
class WatchConfig:
    load_mode: str      # "uniform" | "region"
    fps: float          # → 直接映射为 VideoMetadata.fps
    spatial_token_rate: SpatialTokenRate  # → 映射为 media_resolution
    regions: List[Tuple[float, float]]    # → 映射为 startOffset/endOffset
```

这整个 schema 都是基于 Gemini 的三维控制模型设计的（时间采样率 + 时间范围 + 空间分辨率）。换成千问，`fps` 不能直接传给模型，而要变成"每隔 X 帧抽一帧"的客户端逻辑。

## 三、如果真要迁移到千问 VL，需要改什么？

```
需要改造的层                    工作量       说明
──────────────────────────────────────────────────────────
① SDK 替换                      中等     google-genai → dashscope/transformers
② 视频解码管线（新增）          大        新增 OpenCV/ffmpeg 客户端抽帧逻辑
③ create_video_part() 重写      大        Blob+VideoMetadata → 图片数组
④ infer_on_video() 重写         大        contents 格式完全不同
⑤ media_resolution 逻辑重写     中等     改为控制抽帧图片的 resize 尺寸
⑥ FPS 钳制逻辑调整              小       逻辑类似，但执行位置从服务端→客户端
⑦ Planner prompt 调整           中等     WatchConfig 的参数语义变了
⑧ 长视频压缩策略调整            中等     千问有 token 数限制，策略不同
⑨ 新增依赖                      小       opencv-python, Pillow 等
```

**保守估计：至少 60-70% 的核心代码需要重写。**

## 四、更现实的方案

如果你想用千问或其他模型，有几个思路：

### 方案 A：OpenAI 兼容接口（最省力）
如果千问提供了 **OpenAI 兼容的 API 端点**（比如通过 DashScope 或 vLLM 部署），AthenaQA 已经预留了 `base_url` 参数：
```python
GeminiClient(
    base_url="https://your-qwen-endpoint/v1",  # 自定义端点
    api_key="your-key",
    model="qwen3.5-vl-400b",
)
```
**但这只解决了文本调用的兼容**，视频传入方式仍然不兼容（Gemini 的 `Part/Blob/VideoMetadata` 协议和 OpenAI 的 `messages` 格式完全不同）。

### 方案 B：写一个适配层（中等工作量）
保持 AthenaQA 上层逻辑不变，写一个 `QwenClient` 类替代 `GeminiClient`：
- 在 `create_video_part()` 里用 OpenCV 抽帧，返回图片列表
- 在 `infer_on_video()` 里把图片列表拼成千问 VL 的 messages 格式
- 在 FPS/分辨率控制处，改为客户端逻辑

### 方案 C：等千问支持原生视频 API（最轻松但不确定时间）
目前千问 VL **不支持**直接传视频文件字节。如果未来阿里推出类似 Gemini 的原生视频 API，改造成本会大幅降低。

## 五、总结

| 问题 | 回答 |
|------|------|
| 直接换模型名就能用？ | ❌ 不行 |
| 根本原因 | AthenaQA 深度绑定了 Gemini 的 **SDK 类型**（Part/Blob/VideoMetadata）和**服务端视频采样**机制，千问 VL 没有这些 |
| 最大差异 | Gemini = 传视频字节 + 服务端采样；千问 VL = 客户端抽帧 + 传图片数组 |
| 迁移工作量 | 60-70% 核心代码需重写 |
| 最现实方案 | 写一个 `QwenClient` 适配层，保持上层 Planner/Observer/Reflector 逻辑不变 |
